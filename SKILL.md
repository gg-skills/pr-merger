# GG → Pr Merger → PR Merger

## Overview

Use this skill when the user points at two or more PRs that target the same problem (typically
auto-generated by tooling like t3, or re-rolled human attempts), and asks you to converge them
to a single merged outcome. The role you play is **arbiter / merger**: you read all the
candidates, you do not write competing implementations yourself, and your job is to land the
best parts of each candidate in a single merged result.

The skill also covers the follow-up shape: a single PR (often a follow-up against an
already-merged sibling) where the user wants you to verify whether anything in it is worth
porting to main before closing it.

## Common Misconceptions

| # | Misconception | Correction | Key concept |
|---|---------------|------------|-------------|
| 1 | The merger writes competing implementations | The merger reads candidates and converges; never writes competing code | Arbiter role |
| 2 | Any PR can be picked as winner without comparison | All PRs must be compared head-to-head before picking | Systematic comparison |
| 3 | Cherry-pick check is optional | Always verify if PR has contributions before closing | No orphan wins |
| 4 | Follow-up checks are not needed after merge | New PRs may reference recently merged work | Continuous check |
| 5 | All PRs targeting same problem are siblings | Some may be re-rolls of previous attempts | History awareness |
| 6 | Phase 6 calibration is optional | Must verify "nothing" claim before closing | Verification gate |

### When this skill applies

Trigger phrases the user actually uses:

- "compare PRs X, Y, Z"
- "pick a winner" / "head-to-head" / "merge one of these"
- "check if PR N has anything to contribute that we should cherry-pick"
- "verify if there is anything they have that is better that we should manually incorporate, then close them"
- "now check PR N" as a follow-up after a merge
- "now let's continue doing the same process with prs"
- branch names that look like siblings (e.g. `<topic>`, `<topic>-1`, `<topic>-2`)

Two- and three-way comparisons can be handled by a single comparison subagent;
four-or-more PRs benefit from parallel per-PR reviewers + your own synthesis.

## Operating modes

The user invokes the skill in one of these modes — match the mode before acting:

| User cue | Mode | What you do |
|---|---|---|
| "Compare PRs X, Y, Z" | **compare-only** | Phases 1–4. Return the verdict. Ask before Phase 5. |
| "Proceed" after a verdict | **execute** | Phase 5 + Phase 6. |
| "Check if PR N has anything to contribute that we should cherry-pick" | **cherry-pick check** | Focused single-PR comparison against post-merge main. If anything is found, port directly to main via a subagent (skip the PR-merge dance); close the source PR with the cherry-picks cited. |
| "Verify if there is anything they have that is better that we should manually incorporate, then close them" | **rigorous cherry-pick check** | Same as above, but the Phase 6 calibration pass is **mandatory** before declaring "nothing". |
| "Now check PR N" as a follow-up to a recent merge | **follow-up check** | Focused single-PR comparison, port any wins, close. |

## The 6 phases

### Phase 1 — Snapshot

- List PRs with `gh pr list --state all` to confirm the competing set.
- For each PR, fetch metadata:
  ```
  gh pr view <N> --json title,body,additions,deletions,changedFiles,files,headRefName,mergeable,mergeStateStatus
  ```
- Save full diffs to a scratch directory:
  ```
  gh pr diff <N> > /tmp/pr-compare/pr<N>.diff
  ```
- Identify file overlap across PRs — which files are touched by all (the shared core) and
  which are asymmetric (a single PR's unique additions). Asymmetric files often signal
  scope creep or a unique data fix.
- Find pre-staged worktrees: `git worktree list | grep <branch-pattern>`.
- Check for orphan branches with the same prefix that never opened as PRs — common in
  t3-managed repos; surface them as future cleanup.
- Record the current `origin/main` HEAD; you will compare candidates against it later
  (`git show origin/main:<path>` reads post-merge state without modifying the working tree).

### Phase 2 — Parallel deep review

- Dispatch **one code-reviewer subagent per PR**, in parallel via a single message with
  multiple Agent / Task tool calls.
- Each subagent's prompt must include: the PR number, branch, worktree path, saved diff
  path, paths to baseline files saved in scratch, the repo's `AGENTS.md`/`CLAUDE.md`
  conventions, and a **specific list of things to verify**:
  - algorithm/design summary
  - correctness analysis
  - test coverage assessment
  - top 3 bugs/weaknesses with **file:line precision**
- Ask each reviewer for an explicit verdict: `merge / merge-with-changes / do-not-merge`.

#### Calibration warning — reviewer subagents over-claim

Code-reviewer subagents systematically **over-claim and under-execute**. They will
frequently *say* they ran tests when they did not, or claim a bug exists in only one PR
when the same bug exists in all of them. **Always treat their headline assertions as
hypotheses, not facts.** Phase 3 exists to verify them.

### Phase 3 — Verify load-bearing claims yourself

For every reviewer's "Bug 1 confidence X" claim, confirm directly:

- Run the repo's typecheck script (e.g. `npm run typecheck` or a package-scoped
  equivalent like `npm run typecheck:profiles`) in **each worktree**, not just the
  suspected winner.
- Run the repo's relevant test suites in each worktree (e.g. `npm run test`,
  `npm run test:queue`, `npm run test:profiles`, or whatever the project exposes).
- For asymmetric tests, run them too — drift checks, benchmarks, integration smoke
  tests. Capture exit codes and result counts.
- For each claimed bug: grep the relevant function on each candidate PR. **A bug is
  only a differentiator if it exists on some candidates and not others.** If it
  exists on all (or on main), it is a shared limitation, not a reason to prefer one
  PR over another.
- If a worktree fails because `node_modules/@types/node` (or similar dep) is missing,
  run `npm install` (or the project equivalent) there first.
- For tests against post-merge main, use `git show origin/main:<path>` rather than
  checking out a different branch in your working tree.

### Phase 4 — Synthesize the verdict

- Distinguish **architectural convergence** (multiple PRs implementing the same
  algorithm with cosmetic naming differences) from **algorithmic divergence**
  (genuinely different approaches). The former is the common case for auto-generated
  PR families.
- Identify **shared limitations** present in every candidate (and possibly on main) —
  these are common-future-work items, not differentiators. State them explicitly so
  they don't become hidden debt.
- Pick a winner based on **real correctness wins** (verified bugs in losers the
  winner avoids), not just test count or PR size. The smallest PR is often the most
  fragile; the largest is often over-rewriting.
- List specific **cherry-pick candidates from losers** by file:line. Common candidates:
  - failure-mode tests the winner lacks
  - helper exports
  - small-but-useful npm scripts
  - doc references
  - regression tests for shared limitations
- **Be honest about the winner's bugs**: if a loser's test exposes a bug that the
  winner has too, port the fix from the loser. The winner can still win and the
  cherry-pick improves the merged outcome.

### Phase 5 — Execute

Apply cherry-picks **before merging**, on the winner's branch:

- Edit files in the winner's worktree.
- Run typecheck + tests + the relevant feature-specific commands (drift check,
  benchmarks). Do not commit a partial state.
- If a cherry-picked test fails, that is a real bug discovery — fix it on the
  winner's branch. The cherry-pick was load-bearing, not cosmetic.
- Commit with a HEREDOC message:
  ```
  git -c commit.gpgsign=false commit -m "$(cat <<'EOF'
  <subject line>

  <body describing what was cherry-picked and why>

  Co-Authored-By: ...
  EOF
  )"
  ```
  The message should describe what was cherry-picked and why, and end with the
  standard `Co-Authored-By:` trailer.
- `git push origin <branch>`.

Merge the winner:

```
gh pr merge <N> --squash --delete-branch \
  --subject "..." \
  --body "..."
```

The body should cite the merged commit content **plus any cherry-picks taken from
losers**.

Close losers in parallel — backgrounded `gh pr close` calls then `wait`:

```
gh pr close <X> --comment "..." &
gh pr close <Y> --comment "..." &
wait
```

Each close comment must cite **specific bugs / inferiority reasons** at file:line
precision and any cherry-picks taken from that PR. Vague "superseded by #N" reasons
are not enough — operators reading the closed PR weeks later need to know why it
lost.

The local-branch-delete-blocked error from worktrees holding the branches is
expected. Note it in the final summary; do not try to force.

### Phase 6 — Calibrate

- If the verdict was "nothing worth porting", do **one more rigorous pass** with a
  focused subagent before locking in. The first-pass test-name comparison is often
  misleading; the rigorous pass diffs implementations side-by-side and reads test
  bodies. **This step has produced real cherry-picks on every round where it was
  applied.** Build it into your habit.
- Track shared limitations as future-work items. Surface them in the final summary
  with concrete file:line so the user can either spawn follow-up work or accept them
  as documented.

## Tactical patterns

### gh CLI

- `gh pr diff <N> -- <pathspec>` filters the diff. Useful when corpus data files
  dominate.
- `gh pr view <N> --json files` lists changed files as JSON.
- `gh pr list --state all` confirms the competing set, including already-merged or
  already-closed siblings.
- `gh pr merge <N> --squash --delete-branch --subject "..." --body "..."` is the
  standard merge invocation; the body is where you cite the cherry-picks taken.
- `gh pr close <N> --comment "..."` is the standard loser close; the comment is the
  audit trail.

### Worktree handling

- Pre-staged worktrees enable parallel test runs. `git worktree list | grep
  <branch-pattern>` finds them.
- Run typecheck and tests **in each worktree**, not just the suspected winner. A
  worktree failing on missing deps means `npm install` (or project equivalent) in
  that worktree first.
- For tests against post-merge main without disturbing your working tree, use
  `git show origin/main:<path>`.
- The local-branch-delete-blocked error after merge is expected when a worktree
  still holds the branch. Note it in the summary; do not force.

### Parallel-close pattern

For closing losers in parallel:

```
gh pr close <X> --comment "<specific reason>" &
gh pr close <Y> --comment "<specific reason>" &
wait
```

This is faster than sequential closes and the operations are independent.

### HEREDOC commit messages

Cherry-pick commits and merge commit bodies must use HEREDOC to preserve
formatting:

```
git -c commit.gpgsign=false commit -m "$(cat <<'EOF'
<subject>

<body>

Co-Authored-By: ...
EOF
)"
```

The same HEREDOC shape works for `gh pr merge --body "$(cat <<'EOF' ... EOF)"` and
`gh pr close --comment "$(cat <<'EOF' ... EOF)"`.

### Diff intersection helpers

- `comm -23 a b` and `comm -12 a b` for set-difference and set-intersection on
  sorted line lists. Useful for "which test names are in PR #N but not in main".
- For 3+ PRs with similar diffs, parallel single-PR reviewers + your own
  verification beats a single comparison subagent (which usually runs out of
  context).
- For 2 PRs (or one PR vs already-merged main), a single comparison subagent with
  both files saved to scratch is more efficient.

### Scoped test runs

Repo-specific test scripts (e.g. `npm run test:queue`, `npm run test:profiles`,
or whatever the package exposes) are usually fastest. Fall back to a direct test
runner against a specific file (e.g. `npx tsx --test path/to/specific/test.ts`)
for narrowly-scoped suites.

## Calibration warnings

These are the failure modes of this skill. Internalize them.

### Reviewer subagents over-claim

Code-reviewer subagents will **say** they ran tests when they did not. They will
claim a bug exists in only one PR when the same bug exists in all of them. They
will rank style preferences as bugs. Always re-run their headline assertions
yourself before incorporating into the verdict. The reviewer's job is to surface
hypotheses; **your job is to verify them** before they become the verdict.

### "Nothing to port" feels right but is often wrong

If the verdict was "nothing worth porting", do **one more rigorous pass** with a
focused subagent before locking in. The first-pass test-name comparison is often
misleading — the rigorous pass diffs implementations side-by-side and reads test
bodies. This calibration has produced real cherry-picks on rounds where it was
applied. **Make it a habit, not an afterthought.**

### Common-limitation traps

If every candidate has the same bug, that is **not** a reason to pick one over
another — it is a shared future-work item. Confirm by grepping the relevant
function on every candidate (and on main). Only differentiating bugs count
toward the verdict.

### Smallest is often most fragile; largest is often over-rewriting

PR size is **not** a signal of quality. The smallest PR may have skipped real
edge cases; the largest may have rewritten things that did not need rewriting.
Pick on **verified correctness wins**, not on diff stats.

### Be specific in close comments

Future-you (and other operators reading the closed PR weeks later) will thank
present-you for citing the exact bug at the exact file:line. Vague "superseded
by #N" comments destroy the audit trail.

## Gotchas

Concrete, environment-specific facts that defy reasonable assumptions. Each one
corrects a mistake the agent will make in a PR-merging session without being
told otherwise.

1. **A loser's commits may already be on main via the winner.** When the same
   tooling (t3, re-rolls) generated the sibling PRs, whole subtrees overlap. If
   you cherry-pick blindly, you get empty commits or — worse — silent
   double-applies that pass tests but obscure provenance. **Before any
   `git cherry-pick`, run `git log <loser-sha>..HEAD -- <paths>` in the
   winner's worktree.** If the path-scoped output is empty, the change is
   already there; skip the cherry-pick and cite the duplicate-content reason
   in the loser's close comment.

2. **Use squash-merge for the winner, but `git cherry-pick -x` (not merge) for
   porting loser commits.** `gh pr merge --squash` collapses the winner's
   history into one commit on main, which is what you want for the audit trail.
   But pulling pieces from losers must be cherry-picks (with `-x` to leave the
   `(cherry picked from commit <sha>)` backlink), not merge commits — a merge
   commit from a loser branch drags in unrelated commits and confuses the
   single-author-per-commit assumption squashing depends on.

3. **Never force-push to a loser's feature branch as "cleanup".** Force-pushing
   after the PR is closed loses the review history that the close comment
   references at file:line. Reviewers reading the closed PR weeks later will
   see "this comment was made on an earlier version of this file" and the
   audit trail breaks. Leave loser branches alone — `gh pr close` does not
   delete the branch on GitHub by default, and the worktree-bound
   delete-blocked error locally is expected.

4. **`gh pr list` hides drafts by default.** When confirming the competing PR
   set in Phase 1, `gh pr list --state all` still omits drafts unless you pass
   `--draft` or `--state all --search "draft:true"`. A draft sibling PR that
   no one mentioned but that contains the cleanest implementation is the
   classic miss. Run `gh pr list --search "is:pr <topic>" --state all` (with
   no `--draft` filter) to be sure.

5. **Conflict markers in a loser's diff may be pre-existing, not from your
   cherry-pick.** If `gh pr diff <N>` shows `<<<<<<<` / `=======` / `>>>>>>>`
   blocks, the loser branch is itself stale from an unfinished rebase upstream
   — it was never resolvable to begin with. Don't try to cherry-pick from it;
   close the PR citing "unresolved rebase conflicts on the branch itself" and
   move on. A `|||||||` ancestor line alongside the usual markers is the
   diagnostic giveaway (three-way merge artifact).

6. **Mismatched base branches make PRs look comparable when they aren't.**
   Sibling PRs targeting different base branches (one against `main`, one
   against a feature branch that hasn't merged yet) will show wildly
   different diffs even for the same underlying work. Always check
   `headRefName` **and** `baseRefName` in the Phase 1 `gh pr view --json`
   call. If bases differ, rebase the odd one onto the common base before
   diffing — or your "asymmetric files" analysis will be noise.

## Key principles

- **Trust but verify.** Reviewer subagents over-claim. Re-run their headline
  assertions yourself.
- **Common limitations aren't differentiators.** Same bug on every candidate is a
  future-work item, not a tiebreaker.
- **Cherry-picks compound.** Small wins from each loser improve the merged
  winner. The work is cheap once the worktrees are pre-staged.
- **If "nothing" feels right, one more rigorous pass.** This has produced real
  wins on every round where it was applied.
- **Be specific in close comments.** Cite the exact bug at the exact file:line.

## Repo agnosticism

This skill is designed to work in any repo that uses:

- GitHub PRs (`gh` CLI for fetch + merge + close)
- A typecheck/test surface invokable via npm scripts (or equivalent — Cargo, Go,
  Make targets, etc.)
- An `AGENTS.md` / `CLAUDE.md` / similar guide for repo conventions
- Git worktrees (optional but enables parallel test runs)

Adapt the specific commands (typecheck script name, test script names, drift
check, benchmark commands, etc.) per the repo's own `AGENTS.md`. Where this
skill cites a script like `npm run typecheck:profiles`, treat it as **one
example of "the repo's typecheck script"** — substitute the equivalent for the
repo at hand.

The phases and principles are universal.

## PR Merger Quality Checklist

Use this checklist before delivering any PR merger verdict.

| # | Checklist Item | Why It Matters | Gate |
|---|---------------|---------------|------|
| 1 | **PRs confirmed** — Competing set identified via `gh pr list` | Prevents missing PRs | Phase 1 |
| 2 | **Metadata fetched** — Title, body, diffs, files for all PRs | Enables comparison | Phase 1 |
| 3 | **File overlap identified** — Shared core vs asymmetric files mapped | Shows scope | Phase 2 |
| 4 | **Head-to-head verified** — All PRs compared systematically | Prevents bias | Phase 3 |
| 5 | **Winner based on real wins** — Correctness wins, not test count | Ensures quality | Phase 4 |
| 6 | **Cherry-picks listed** — Specific file:line from losers | Preserves value | Phase 4 |
| 7 | **Cherry-picks applied before merge** — Applied to winner's branch | Ensures integration | Phase 5 |
| 8 | **Tests pass after cherry-picks** — Typecheck + tests run | Verifies cherry-picks | Phase 5 |
| 9 | **Phase 6 calibration done** — "Nothing" claim verified | Prevents orphan wins | Phase 6 |
| 10 | **Losers closed with reasons** — Explicit close messages | Documents decision | Closeout |

### Quality Tiers

| Tier | Criteria | Use When |
|------|----------|----------|
| **Minimal** | Items 1-5, 10 | Simple two-PR comparison |
| **Standard** | Items 1-8, 10 | Standard merge with cherry-picks |
| **Full** | All 10 items | Complex multi-PR with rigorous check |

### Pre-Verdict Verification

```
□ All competing PRs identified via gh pr list
□ Metadata fetched for each PR
□ File overlap mapped (shared core vs asymmetric)
□ Head-to-head comparison complete
□ Winner based on real correctness wins
□ Cherry-pick candidates listed
□ Cherry-picks applied and tested
□ Losers closed with explicit reasons
```

## PR Merger Consistency Validator

Before finalizing, verify:

### Consistency Check Matrix

| Check | What to Verify | How to Fix |
|-------|---------------|------------|
| **Winner vs Evidence** | Winner has real correctness wins | Re-compare |
| **Cherry-picks vs Losers** | Cherry-picks from losers only | Verify origin |
| **Tests vs Cherry-picks** | All tests pass after cherry-picks | Fix cherry-picks |
| **Close vs Verdict** | Close messages match verdict reasons | Update messages |

### Red Flags (Never Present)

- [ ] Winner chosen without head-to-head comparison
- [ ] Cherry-picks from winner to winner
- [ ] Tests not run after cherry-picks
- [ ] Losers closed without reasons
- [ ] "Nothing" claim without Phase 6 verification
